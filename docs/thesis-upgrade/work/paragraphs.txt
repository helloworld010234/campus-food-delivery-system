136	Heading 2	2.1 校园餐饮服务数字化研究
137	Normal	校园餐饮服务数字化的核心不是把线下菜单简单搬到线上，而是把需求产生、商品选择、订单生成、商户处理和结果反馈整合成连续的信息链路。学生端需要快速找到可服务商户和可购买菜品，商户端需要及时处理订单并维护营业状态，管理端需要对经营数据和异常状态进行监督。若系统仅具备展示页面而缺乏订单状态和数据隔离机制，就难以支撑真实校园场景下的多角色协作。
138	Normal	从用户接受角度看，系统是否有用、是否易用会影响使用意愿[1]。在校园场景中，有用性体现在减少排队和沟通成本，易用性体现在小程序页面路径短、商户切换清晰、购物车状态明确、下单后状态可见。信息系统成功模型进一步指出，系统质量和信息质量会影响使用效果[8]。因此，论文在评价系统时不能只说“功能丰富”，还要从信息准确、状态一致、接口稳定和服务反馈等角度说明质量。
139	Normal	校园系统还具有管理属性。用户并非完全匿名社会消费者，商户也不是开放平台中的任意经营者，系统需要把校园、商户、员工和订单纳入同一数据边界中。项目中的多商户迁移脚本、商户管理接口和商户作用域参数，正是为了适配这一管理属性。后续章节将把该属性转化为需求约束和数据模型设计，而不是把它作为附加功能简单带过。
140	Normal	校园餐饮数字化的启示在于，系统设计必须同时服务学生体验和管理秩序。后续需求分析因此把用户端便利性、商户端处理效率和平台端数据沉淀并列考虑。
141	Normal	因此，本系统的价值不能简单表述为“线上点餐更方便”，而应落到可验证的服务节点上：商户列表是否能反映服务入口，菜品状态是否能减少无效下单，购物车是否能区分商户上下文，订单状态是否能被用户和商户共同追踪。这些节点共同构成校园餐饮数字化的最小闭环。
142	Heading 2	2.2 在线点餐与外卖应用研究
143	Normal	在线点餐研究普遍认为，用户体验不仅来自下单动作本身，还来自完整服务过程的确定性。用户在点餐前关注商户是否营业、菜品是否可售、价格和配送费是否清楚；下单时关注地址、备注、数量和金额是否正确；下单后关注支付状态、订单状态和配送时间是否明确。相关研究从体验、满意度和持续使用角度解释了外卖应用的价值[3][4][9]，这些结论可以转化为校园系统的需求表达。
144	Normal	外卖应用在不同人群中的使用感知也存在差异[10]。校园用户的差异可能来自年级、宿舍位置、作息时间、消费习惯和对配送速度的敏感程度。由于本课题没有开展正式问卷调查，论文不对用户群体差异作统计结论，但可以据此说明系统设计应保持清晰的信息呈现和稳定的业务流程，避免让用户在商户切换、购物车和订单状态之间产生歧义。
145	Normal	本系统面向校园外卖点餐，不直接研究平台推荐算法、骑手路径优化或商家竞价排序。其研究重点是工程实现层面的业务闭环：用户在小程序端完成商户选择、菜品浏览、购物车操作和订单提交；商户或管理员在 Web 端处理订单、维护商品和查看统计；后端在数据库中维护订单、明细、商户作用域和状态流转。这一定位使论文论证更贴近项目事实。
146	Normal	在线点餐研究对本系统的直接启示是：下单入口、订单状态和服务反馈会共同影响用户体验。因此后续实现不仅分析提交订单，还会分析支付边界、订单通知和历史订单等反馈环节。
147	Heading 2	2.3 系统关键技术基础
148	Normal	Spring Boot 为后端服务提供了约定优于配置的工程组织方式，适合构建包含控制器、服务层、持久层和配置组件的 Web 应用[11]。本系统后端采用多模块 Maven 项目，将通用工具、实体 DTO/VO 和服务端实现拆分到不同模块，有利于降低代码耦合。MyBatis 与 Spring Boot 的集成用于完成 SQL 映射和动态查询[12]，使订单、菜品、购物车等实体能够围绕业务条件组织数据库访问。
149	Normal	MySQL 负责存储用户、商户、菜品、套餐、购物车、订单和订单明细等结构化数据[13]。Redis 更适合缓存菜品、套餐、店铺状态等高频读取数据，并辅助处理需要快速访问的状态信息[14]。JWT 用于在无状态接口中携带用户身份和权限声明[15]，项目中管理端和用户端分别使用不同 token 名称，便于区分请求来源。WebSocket 支持服务端向客户端发送实时消息[16]，系统中的新订单和催单通知由此获得更及时的反馈通道。
150	Normal	小程序端采用 uni-app 开发，能够以 Vue 风格组织页面、组件和接口请求，并面向微信小程序运行环境输出应用[17][18]。项目中的 pages.json 表明，小程序端包含首页、下单、商户切换、订单详情、支付、地址管理、个人中心和历史订单等页面；request.js 和 merchant.js 则封装了请求头与商户作用域参数。管理端使用静态 HTML/JS 应用部署在 Nginx 目录中，通过统一 API 服务访问后端。
151	Normal	这些技术并非孤立堆叠：Spring Boot 支撑后端分层，MyBatis 连接业务条件与 SQL，MySQL 保存事务型订单数据，Redis 缓存菜品、套餐等高频读取数据，JWT 处理多端鉴权，WebSocket 支撑订单提醒。
152	Heading 2	2.4 技术选型与适配性分析
153	Normal	本系统选择 Spring Boot、MyBatis 和 MySQL 作为后端主干，主要原因在于校园点餐业务以事务型数据处理为核心，订单提交、购物车清理、订单明细写入和状态变更都需要清晰的数据一致性边界。关系数据库能够表达用户、商户、菜品、订单之间的结构化关系，MyBatis 便于控制多条件查询和商户作用域过滤，Spring Boot 则提供统一的 Web 服务框架与配置管理能力。
154	Normal	前端选择微信小程序和 Web 管理端组合，符合校园用户和商户管理的使用差异。学生侧更适合轻量入口，不需要额外安装应用；商户或管理员侧需要更大的信息展示空间和更复杂的表单操作，因此采用 Web 管理端更合适。uni-app 对小程序页面、组件和请求封装具有较高适配性，静态 Web 管理端部署在 Nginx 下，也降低了运维复杂度。
155	Normal	多商户扩展是本系统技术选型中的重要约束。若仅采用单店表结构，购物车、分类、菜品和订单容易在商户之间产生混用风险；加入 merchant_id、campus_id、account_type 等字段后，系统可以在服务层和 Mapper 层进行作用域约束。项目中同时保留 schema 检测与兼容回落逻辑，说明系统设计考虑了迁移过程中的数据库状态差异。
156	Normal	由此可见，技术选型服务于校园点餐的具体约束：事务数据用关系数据库表达，多端访问用统一接口承接，商户隔离由作用域字段和查询条件实现，实时提醒由长连接补充。
157	Normal	本章从外卖应用研究、校园服务数字化和关键技术三个层面建立分析依据，为第三章将场景问题转化为系统需求提供支撑。

160	Heading 2	3.1 用户角色与业务边界
161	Normal	系统主要角色包括学生用户、商户员工、商户管理员和平台管理员。学生用户通过小程序完成商户选择、菜品浏览、购物车维护、地址管理、订单提交、支付操作、历史订单查看和催单。商户员工侧重订单处理、营业状态维护和菜品信息查看。商户管理员在此基础上还需要维护菜品、分类、套餐、员工和商户基础信息。平台管理员则关注跨商户的账号、商户状态和运营数据。
162	Normal	角色边界的核心是权限和数据范围。用户端只能访问自己的地址、购物车和订单；商户端只能处理本商户相关的菜品、订单和统计数据；平台管理员可进行更高范围的商户管理。项目中的 account_type、merchant_id、JWT 拦截器和商户作用域工具类为这种边界提供了实现基础。需求建模时必须明确边界，否则后续数据模型和接口设计无法解释为什么需要多商户字段。
163	Normal	表3.1 概括了主要角色和需求边界，表中内容来自项目页面、控制器和前端接口封装。由于项目并未提供正式问卷数据，本章不对用户需求作统计推断，而以系统角色和业务流程作为需求来源。
164	Normal	表3.1 主要角色与需求边界
166	Normal	角色边界决定了权限设计的基本方向。学生只关注个人订单和地址，商户关注所属商户数据，平台管理员关注全局配置和异常处理，这一差异将直接影响接口权限和数据过滤。
167	Heading 2	3.2 核心业务流程
168	Normal	学生点餐流程从进入小程序首页开始。用户首先确认当前商户，如果需要切换则进入商户列表页，选择目标商户后返回菜品展示页面。随后用户按分类浏览菜品或套餐，选择口味和数量后加入购物车。购物车中的商品必须与当前商户保持一致，避免用户在不同商户之间混合下单。确认地址、备注、金额和配送信息后，系统生成订单并清理相应商户范围内的购物车数据。
169	Normal	商户处理流程从订单通知开始。订单支付成功或进入待确认状态后，后端通过 WebSocket 向商户端或平台管理员发送新订单消息。商户可以确认订单、拒单、派送、完成或取消订单。订单状态变化需要在数据库中持久化，并在用户端详情页和历史订单页中体现。若用户发起催单，后端同样通过 WebSocket 将提醒发送到对应商户或管理员侧。
170	Normal	运营管理流程围绕商户、商品和报表展开。管理端接口包括分类、菜品、套餐、订单、工作台、报表、店铺和商户等模块。商户维护菜品和套餐信息，平台或商户管理人员查看订单统计、营业额统计和销量排行。该流程的需求重点是数据口径一致，尤其在多商户场景下，订单数量、营业额和销量排行等订单派生统计应根据商户作用域过滤；用户统计若保留平台级口径，则需要在界面或文档中明确说明。
171	Normal	流程建模的重点是状态变化。商户切换改变商品范围，购物车承接下单前状态，订单提交固化业务结果，支付与通知推动后续处理；这些状态共同构成系统验证的主线。
172	Normal	从建模角度看，流程之间还存在顺序依赖。商户切换必须先于菜品浏览和购物车读取，购物车确认必须先于订单生成，支付状态变化必须先于新订单通知，订单状态沉淀又为历史订单和报表统计提供数据来源。若任一顺序被打破，系统表面仍可能显示页面，但业务语义已经不完整。
173	Heading 2	3.3 功能需求分析
174	Normal	用户端功能需求包括登录认证、商户列表、商户切换、分类浏览、菜品详情、口味选择、购物车增减、购物车清空、地址管理、订单提交、支付处理、订单详情、历史订单、再来一单和催单。小程序页面结构已经覆盖这些功能入口，后端用户端控制器也提供了地址、分类、菜品、套餐、购物车、订单和用户相关接口。功能设计应保证页面入口、请求参数和后端业务对象之间保持一致。
175	Normal	管理端功能需求包括员工登录、分类管理、菜品管理、套餐管理、订单管理、工作台概览、报表统计、店铺营业状态和商户管理。项目中的 api-service.js 对这些接口进行了集中封装，说明管理端不是单一订单列表，而是面向商户经营与平台管理的综合后台。需求分析时应区分“平台级管理”和“商户级管理”，避免所有管理功能都默认拥有全局数据权限。
176	Normal	多商户功能需求是本系统区别于普通单店点餐系统的重要内容。系统需要支持商户基础信息维护、商户营业状态调整、用户端商户切换、商品数据按商户展示、购物车按商户隔离、订单按商户归属和统计按商户过滤。多商户迁移脚本和 Mapper 条件已经反映出这些需求，因此后续设计要把 merchant_id 作为业务范围控制字段，而不仅是数据库中的普通属性。
177	Normal	功能需求并不只是页面入口清单，还必须说明功能之间的依赖关系。没有商户切换，商品展示和购物车隔离缺少上下文；没有订单状态，历史订单和管理端处理就无法形成闭环。
178	Heading 2	3.4 非功能需求分析
179	Normal	可用性需求要求学生端操作路径尽量短，关键信息清楚可见。用户进入首页后应能识别当前商户、分类、菜品价格、起送或配送信息，并能在提交订单前确认地址和金额。对商户端而言，订单状态和操作按钮应保持一致，避免商户因界面状态不清而重复确认或误取消订单。移动点餐研究中对满意度和持续使用的讨论，也说明稳定、清楚的交互对系统价值具有重要影响[3][9]。
180	Normal	安全性需求主要体现在身份认证、权限边界和个人信息保护。JWT 用于识别用户身份，管理端与用户端使用不同请求头名称，有助于区分访问来源。地址、电话、订单明细等信息涉及个人信息，应遵循最小必要原则，避免无关角色访问用户隐私数据[19]。同时，系统需要在服务层或查询层保证商户数据隔离，防止商户员工查询其他商户订单。
181	Normal	可靠性和可维护性需求要求系统在数据库迁移和运行环境差异下仍能保持基本可用。项目中存在多商户 schema 检测与兼容回落逻辑，说明系统考虑了旧库与新库并存的过渡情况。软件质量模型强调功能适合性、可靠性、易用性、维护性和安全性等质量属性[20]，本系统验证也应围绕这些质量属性展开，而不能只统计接口数量。
182	Normal	非功能需求为实现质量设定底线。身份认证、个人信息保护、数据隔离和异常兼容不一定直接体现在界面上，却决定系统能否在校园多角色场景中稳定运行。
183	Normal	可维护性还要求论文把兼容逻辑写清楚。项目中既有单店基础结构，也有多商户迁移后的结构，服务层通过 schema 检测和字段兼容处理不同数据库状态。该做法降低了迁移风险，但也意味着测试报告需要注明所处数据结构，不能把某一环境下的结果直接推广到所有部署环境。

191	Heading 1	第四章 系统架构与数据模型设计
192	Heading 2	4.1 总体架构设计
193	Normal	系统采用前后端分离的总体架构，用户端、商户/管理端、后端服务和数据库分别承担不同职责。用户端以微信小程序形式运行，负责学生侧交互；商户/管理端以 Web 静态应用形式部署，负责订单处理、商品维护和运营统计；后端 Spring Boot 服务提供统一 API；MySQL 存储核心业务数据，Redis 辅助处理缓存或状态数据，WebSocket 用于订单消息推送。
194	Normal	从网络化软件架构角度看，系统通过清晰的接口边界连接不同客户端和后端资源，符合面向资源和分层处理的基本思想[21]。小程序端和 Web 管理端并不直接访问数据库，而是通过 REST 风格接口与后端服务交互。后端服务再通过 Mapper 层访问数据库，使业务规则、权限校验和数据持久化之间形成分层结构。
195	Normal	架构设计需要兼顾当前功能和后续扩展。由于校园餐饮商户规模通常可控，系统无需一开始就引入复杂微服务拆分；但多商户作用域、订单通知和报表统计等模块具有较强扩展需求，因此在单体 Spring Boot 架构内部保持模块边界清晰更为合适。软件架构实践强调质量属性驱动的设计权衡[22]，本系统在复杂度、可维护性和部署成本之间选择了较稳妥的分层单体方案。
196	Normal	总体架构的取舍是以毕业设计规模和校园业务闭环为前提：系统保持单体后端和多端前端组合，既能降低部署复杂度，也能通过模块划分保留后续扩展空间。
197	Heading 2	4.2 前后端分离设计
198	Normal	小程序端和管理端通过 HTTP 接口访问后端服务。用户端请求携带 authentication，管理端请求携带 token，后端通过对应拦截器解析身份信息。这样的设计使同一后端能够同时服务移动端和 Web 端，并根据访问来源和账号类型执行不同权限规则。前端页面只负责展示和交互，核心业务校验仍由后端完成。
199	Normal	小程序端页面包括首页、订单提交、商户切换、订单详情、支付、地址、个人中心和历史订单等。页面之间通过商户上下文、购物车状态和订单号形成业务链路。管理端 API 封装包括菜品、分类、套餐、订单、工作台、报表、店铺和商户等模块，使前端页面能够以统一方式发起请求。前后端分离降低了页面变更对后端业务规则的影响，也方便后续替换或扩展客户端形态。
200	Normal	接口参数设计中，merchantId 和 shopId 的兼容封装具有实际意义。旧版本或部分页面可能使用 shopId 表示店铺，新的多商户逻辑更倾向于使用 merchantId。小程序工具类同时传递两类参数，后端服务再统一解析请求商户，可以减轻迁移期间前后端字段不一致带来的风险。这一设计体现了系统从单店模型向多商户模型过渡的工程特征。
201	Normal	前后端分离的关键收益在于职责清楚。小程序和 Web 管理端可以围绕各自交互优化，后端则集中维护身份、商户范围、订单状态和数据一致性。
202	Heading 2	4.3 后端模块设计
203	Normal	后端按控制器、服务、Mapper 和实体对象组织。控制器层负责接收 /admin 与 /user 请求并返回统一结果；服务层负责业务规则，如订单提交、支付处理、购物车清理、商户校验和状态流转；Mapper 层负责 SQL 执行和多条件查询；实体、DTO 和 VO 分别承担持久化对象、请求对象和响应对象职责。这种分层有利于保持接口、业务和数据访问之间的边界。
204	Normal	核心模块包括用户登录、地址簿、分类、菜品、套餐、购物车、订单、商户、工作台和报表。订单模块是业务中心，它连接用户、地址、购物车、订单明细、支付状态、商户状态和通知消息。购物车模块则承担下单前的临时状态管理，必须根据用户和商户共同定位数据。报表模块基于订单数据生成营业额、订单数量和销量排行等统计结果。
205	Normal	多商户支持模块由 schema 检测、启动守卫、商户作用域工具类和多商户字段共同组成。系统启动时检测数据库是否具备 campus、merchant 以及核心表的商户字段；服务运行时根据账号类型和当前商户解析访问范围；查询执行时通过 Mapper 条件过滤商户数据。该模块不是独立页面，而是贯穿商品、购物车、订单和统计的横向能力。
206	Normal	后端模块划分使订单这一核心流程能够调用用户、地址、购物车、商户、支付和通知等能力，同时又避免所有逻辑挤在控制器层。该设计也便于后续单独检查关键服务。
207	Heading 2	4.4 数据库实体与多商户隔离
208	Normal	数据库基础实体包括用户、地址、分类、菜品、菜品口味、套餐、套餐菜品关系、购物车、订单、订单明细和员工。多商户扩展新增或使用校区、商户实体，并在员工、分类、菜品、套餐、购物车、订单等表中增加商户或校区作用域字段。通过这种设计，系统可以在同一套业务表中区分不同商户的数据归属。
209	Normal	表4.1 概括了核心实体与作用域字段。需要注意的是，主 SQL 文件仍体现单店基础结构，多商户字段来自迁移脚本。因此论文把多商户数据模型描述为在基础表结构上的扩展设计，并在验证章节说明其依赖迁移脚本和实际数据库状态。
210	Normal	表4.1 核心实体与作用域字段
212	Normal	多商户隔离的关键在于业务写入和查询都必须使用同一商户上下文。用户加入购物车时，系统根据请求商户或默认商户解析 merchantId，并在查询购物车时以用户和商户共同过滤。订单提交时，系统从当前商户范围内读取购物车并生成订单，随后删除该商户范围内的购物车数据。管理端查询订单和统计时，也需要根据当前账号可访问商户进行过滤。
213	Normal	数据模型设计的关键不在表数量，而在关系是否能表达业务边界。merchant_id、campus_id 和订单状态字段共同决定商品展示、购物车隔离和订单统计是否可信，附录 A 对主要字段作进一步说明。
214	Normal	多商户隔离不仅是给表增加字段，还要求字段在写入、查询和统计中持续生效。写入阶段要保存订单所属商户，查询阶段要依据当前账号过滤商户范围，统计阶段要把商户范围传递到订单聚合条件中。只有三个阶段同时成立，merchant_id 才真正成为业务边界，而不是形式化字段。
215	Heading 2	4.5 接口边界与安全设计
216	Normal	接口边界主要通过路径前缀、请求头和权限上下文实现。用户端接口集中在 /user 下，面向学生用户；管理端接口集中在 /admin 下，面向员工、商户管理员和平台管理员。JWT 拦截器解析 token 后写入当前用户上下文，服务层再基于上下文判断用户是否有权访问对应订单或商户数据。JWT 的标准机制为无状态身份传递提供了基础[15]。
217	Normal	安全设计需要覆盖身份认证、越权访问和个人信息保护。用户地址、手机号、订单明细等信息不应暴露给无关商户或其他用户；商户员工也不应访问非所属商户的订单。系统通过 merchant_id 查询条件和账号类型区分访问范围，但仍需要在后续完善中持续检查接口是否存在漏加商户过滤的情况。个人信息处理应遵循必要性和目的限定原则[19]。
218	Normal	WebSocket 通知接口用于新订单和催单消息传递。服务端维护 sid 与连接会话的映射，可以向全部客户端或指定客户端发送消息。订单支付成功或用户催单时，系统构造包含事件类型、订单号和商户标识的消息，并推送给商户员工或平台管理员。由于 WebSocket 是长连接机制，后续还需要关注连接失效、重连和消息确认等可靠性问题[16]。
219	Normal	接口安全设计需要和数据模型同时成立。只有 token 识别身份而没有商户过滤，仍可能出现越权；只有商户字段而没有服务层校验，也难以保证请求行为可靠。
220	Normal	接口边界还应处理异常路径。例如用户端请求缺少商户标识时，后端需要使用默认商户或返回明确错误；管理端账号缺少商户归属时，应区分平台管理员和普通商户员工。将这些异常路径纳入设计说明，可以避免论文只描述理想流程，而忽略真实系统中最容易产生越权或数据错配的位置。
221	Normal	本章说明了多端架构、后端模块、数据实体和接口安全边界，第五章将进一步展开这些设计在关键业务流程中的实现方式。
223	Heading 1	第五章 关键业务流程实现
224	Heading 2	5.1 登录认证流程实现
225	Normal	登录认证是系统访问控制的入口。用户端通过小程序登录获取身份凭据，后续请求在 authentication 请求头中携带 token；管理端通过员工账号登录，在 localStorage 中保存 token，并在后续请求头中携带 token。后端拦截器分别处理用户端和管理端请求，将解析出的身份写入当前上下文，为后续订单、购物车和商户权限判断提供依据。
226	Normal	JWT 的优势在于服务端无需为每次请求维护会话状态，令牌中可以携带必要的身份声明[15]。但这并不意味着可以忽略权限校验。系统在解析 token 后仍需要根据用户身份、账号类型和商户归属检查业务对象。例如用户查询订单详情时，若订单不属于当前用户，则应拒绝访问；商户员工处理订单时，若订单不属于其商户，也应拒绝操作。
227	Normal	登录认证流程还需要考虑前端异常。若 token 失效或缺失，前端应引导用户重新登录；若管理端请求被拒绝，应清理本地 token 或提示重新认证。论文不展开具体页面交互细节，但在质量分析中将认证失败、越权访问和 token 缺失作为安全性检查项。
228	Normal	登录认证流程的风险集中在 token 缺失、过期和账号类型识别错误。后续测试应分别验证用户端与管理端请求头名称、未登录拦截和订单访问边界。
229	Heading 2	5.2 商户切换流程实现
230	Normal	商户切换是校园多商户场景中的关键流程。小程序端提供商户列表和切换页面，用户可以选择目标商户后返回首页浏览对应商品。请求封装工具会把当前商户标识作为 merchantId 或兼容字段 shopId 传递给后端。后端服务接收到请求后解析商户标识，并根据商户营业状态、校区状态和默认商户配置确定可用范围。
231	Normal	商户切换不仅影响页面展示，还影响购物车和订单提交。若用户从商户 A 切换到商户 B，页面展示的分类、菜品、套餐应来自商户 B，购物车也应读取用户在商户 B 范围内的数据。否则会出现把商户 A 商品提交到商户 B 订单中的错误。项目中的购物车服务根据用户和商户共同查询购物车，正是为了解决这一问题。
232	Normal	商户切换流程的实现体现了从单店模型向多商户模型迁移的难点。单店系统可以默认所有商品属于同一店铺，而多商户系统必须在用户请求、前端状态、后端服务和数据库查询之间传递一致的商户上下文。该流程的正确性直接影响后续菜品浏览、购物车隔离和订单提交。
233	Normal	商户切换流程的独有风险是前端商户状态与后端解析结果不一致。系统同时传递 merchantId 与兼容字段，可以降低迁移期间字段差异造成的展示和下单错配。
234	Heading 2	5.3 菜品浏览流程实现
235	Normal	菜品浏览流程从分类列表开始。用户进入首页后，系统按当前商户读取分类和商品数据，并在小程序端以菜单形式展示。用户点击分类后可以查看菜品、套餐、价格、图片和口味信息。后端菜品查询需要结合分类、状态和商户作用域，保证展示结果与当前商户一致。
236	Normal	菜品详情与口味选择是影响用户下单体验的重要环节。对于有规格或口味的菜品，用户需要在加入购物车前选择具体选项；对于套餐，系统需要展示套餐包含的菜品信息。若商品已停售或所属商户不一致，后端服务应拒绝加入购物车。这样可以避免前端状态滞后导致不可售商品被下单。
237	Normal	菜品浏览流程还涉及图片路径处理。项目中存在图片解析工具，用于处理商品图片的访问地址。论文不把图片展示作为核心创新，但将其视为商品信息质量的一部分。准确的菜品名称、价格、状态和图片能够降低用户确认成本，从而提高点餐流程的可用性。
238	Normal	菜品浏览流程的正确性依赖商户、分类和商品状态三类条件。若其中任一条件未参与过滤，页面就可能展示不可购买或不属于当前商户的商品。
239	Heading 2	5.4 购物车隔离流程实现
240	Normal	购物车承担下单前的临时状态管理。在单店系统中，购物车通常只需要按用户区分；在多商户系统中，购物车必须同时按用户和商户区分。项目中的购物车服务在新增、查询、减少和清空操作中解析请求商户，并在数据库支持商户字段时写入或查询 merchant_id。这种设计避免了不同商户商品混入同一订单。
241	Normal	加入购物车时，服务层先根据请求中的 merchantId 或 shopId 得到逻辑商户，再判断菜品或套餐是否属于该商户。若商品不存在或商户不匹配，则抛出业务异常；若购物车中已有相同商品和口味，则增加数量；否则新增购物车记录。该流程把前端操作转化为后端可验证的业务规则。
242	Normal	购物车清空和订单提交后的清理同样需要商户范围。用户提交某一商户订单后，只应删除该商户范围内的购物车记录，而不应清空用户在其他商户下暂存的商品。该细节虽然在界面上不显眼，却是多商户点餐系统正确性的关键。表5.1 列出了购物车及相关流程中的主要接口映射。

247	Heading 2	5.5 订单提交流程实现
248	Normal	订单提交是系统业务闭环的核心。用户在确认地址、商品、数量、金额和备注后提交订单，后端根据当前用户和商户读取购物车，校验地址是否存在、商户是否可下单、购物车是否为空，然后生成订单主表和订单明细。订单主表记录用户、地址、金额、状态、支付状态、商户和配送等信息，订单明细记录菜品或套餐的具体内容。
249	Normal	订单提交过程需要保证数据一致性。若订单主表写入成功但订单明细写入失败，或购物车提前清理后订单生成失败，都会导致业务状态异常。因此服务层通常需要事务控制，使订单、明细和购物车清理处于同一业务边界内。项目中的订单服务在提交订单时组织购物车、订单和明细数据，并在完成后清理对应商户购物车，体现了这一事务性要求。
250	Normal	订单编号、金额和配送费等字段也需要在提交时确定。多商户场景下，订单应记录商户名称、商品金额、配送费和商品数量等信息，以便后续展示和统计不依赖实时商户信息。即使商户后续修改名称或配送费，历史订单仍应保留当时的订单信息。该设计有助于提升订单数据的可追溯性。
251	Normal	订单提交的实现重点是事务一致性。项目订单服务在提交方法上使用事务边界组织地址校验、购物车读取、订单主表写入、明细批量写入和购物车清理，任何环节失败都不应留下半完成订单。
252	Heading 2	5.6 支付边界流程实现
253	Normal	支付流程在论文中需要谨慎表述。项目代码中存在支付接口、支付状态和支付成功后的订单状态更新逻辑，也存在模拟支付配置。用户提交订单后，系统进入待支付状态；支付成功后，订单状态变为待商户确认，支付状态变为已支付，并触发新订单通知。该流程可以说明系统具备支付状态处理和支付接口对接能力。
254	Normal	由于缺少真实生产支付凭证和完整上线交易数据，论文不将系统表述为已经完成真实微信支付生产上线。更稳妥的表述是：系统保留微信支付工具与支付回调边界，同时支持模拟支付以便在开发和测试环境中完成订单状态流转验证。这样既反映代码事实，又避免夸大项目能力。
255	Normal	支付边界的清晰表达对系统验证也有帮助。功能测试可以验证订单从待支付到待确认的状态变化、支付后通知是否触发、取消订单时支付状态如何处理；但不能把模拟支付结果等同于第三方支付平台真实结算能力。该边界将在第六章的测试和局限性中继续说明。
256	Normal	支付边界的独有要求是区分状态验证与真实结算能力。开发环境可以用模拟支付验证订单流转，但正式支付还需要证书、回调验签、退款和对账等生产条件。
257	Heading 2	5.7 订单通知流程实现
258	Normal	订单通知用于缩短用户下单后到商户响应之间的信息延迟。系统通过 WebSocket 建立服务端与商户端或管理端之间的长连接，当订单支付成功或用户催单时，后端构造消息并发送给对应客户端。WebSocket 协议支持在单个 TCP 连接上进行双向通信，适合订单提醒这类需要及时送达的场景[16]。
259	Normal	项目中的 WebSocket 服务维护 sid 与连接会话的映射，既可以向全部客户端广播，也可以向指定客户端发送消息。多商户场景下，订单通知按代码实现优先发送给所属商户员工，并同步通知平台管理员；当员工作用域不可用或订单缺少商户标识时，通知会退化为广播。消息中包含事件类型、订单标识、商户标识和提示内容，使前端能够在收到消息后更新订单列表或提醒操作人员处理。
260	Normal	订单通知的可靠性仍有改进空间。当前流程可以实现基本消息推送，但长连接断开、浏览器刷新、网络波动和消息确认机制仍需要后续完善。论文将其定位为实时提醒能力，而不是严格意义上的可靠消息队列。这样能够准确反映系统现状，并为展望章节留下合理改进方向。
261	Normal	通知流程按代码实现发送给所属商户员工，并同步通知平台管理员；在员工作用域不可用或商户标识缺失时退化为广播。该实现满足基础提醒，但还不是可靠消息系统。
262	Normal	在毕业设计验收中，通知流程更适合作为“及时提醒”能力验证，而不是作为严格消息中间件验证。测试时可以记录连接建立、支付后触发、新订单提醒显示和催单提醒显示等现象；对于浏览器刷新、连接断开和离线补偿等问题，则应归入后续可靠性改进，避免把未实现机制包装成既有能力。
263	Heading 2	5.8 历史订单与再来一单流程实现
264	Normal	历史订单功能满足用户查看消费记录和追踪订单状态的需求。小程序端提供历史订单页面，后端按当前用户分页查询订单，并可根据订单状态或支付状态过滤。订单详情页展示订单主信息、配送信息、商户信息和订单明细，使用户能够确认每次点餐的商品、金额和状态。
265	Normal	再来一单功能通过历史订单明细重建购物车。用户选择某个历史订单后，系统读取该订单的明细，将菜品或套餐重新加入当前用户购物车，并保留商户作用域。多商户场景下，该流程尤其需要注意订单所属商户与购物车商户的一致性，否则用户可能把旧订单中的商品加入错误商户的购物车。
266	Normal	历史订单还为运营统计提供基础数据。订单状态、支付状态、商户标识、商品数量和金额等字段不仅用于用户展示，也用于工作台和报表统计。系统在订单提交时保存必要冗余信息，可以减少后续统计对实时商品或商户配置的依赖，提高历史数据解释能力。
267	Normal	再来一单的风险在于历史订单商品可能已经停售或商户状态发生变化。当前流程重点复用历史明细重建购物车，后续可进一步补充商品有效性复核。
268	Heading 2	5.9 运营统计流程实现
269	Normal	运营统计服务于商户经营和平台管理。管理端封装了营业额统计、用户统计、订单统计、销量排行和报表导出等接口，后端通过订单数据计算经营指标。工作台模块还提供今日数据、订单概览、菜品概览和套餐概览，使管理人员能够快速掌握当前经营状态。
270	Normal	多商户统计必须处理数据范围。商户员工或商户管理员查看订单数量、营业额和销量排行时，应只看到所属商户的数据；平台管理员查看时，可以根据权限查看更大范围的数据。项目中的订单派生统计服务根据 schema 和商户作用域解析查询条件，说明相关报表已经考虑多商户过滤。用户统计如果采用平台级口径，则应作为全局数据说明，不能直接当作商户级隔离统计。若忽略这些口径差异，统计结果会混入其他商户订单或被误读为商户自身用户数据，影响管理决策。
271	Normal	报表导出使用 POI 相关依赖和运营数据报表模板，可以将统计结果转化为 Excel 文件。论文不把报表模板本身作为研究重点，而是说明系统具备将订单数据沉淀为运营指标的能力。该能力使校园商户能够从单纯处理订单转向观察销售结构、时段变化和商品表现。
272	Normal	运营统计应区分商户级和平台级口径。订单、营业额、销量排行等订单派生统计适合按商户过滤；用户统计若保留全局口径，则不宜直接等同于商户级隔离统计。
273	Normal	运营统计的可信度取决于指标定义是否稳定。营业额应明确是否只统计已支付或已完成订单，订单数量应明确是否排除取消订单，销量排行应明确按商品数量还是销售金额排序。论文在分析统计模块时采用订单派生数据作为依据，能够减少主观评价，也便于后续通过数据库记录进行复核。
274	Normal	本章围绕认证、商户、商品、购物车、订单、支付、通知和统计分析了核心实现，第六章将从功能和质量角度验证这些流程。

284	Heading 2	6.2 功能测试
285	Normal	登录认证测试检查用户端和管理端 token 传递是否正常。用户端请求应携带 authentication，管理端请求应携带 token。当请求缺少 token 或 token 无效时，后端应拒绝访问受保护接口。通过该测试可以确认系统的基本身份边界，但还需要结合订单详情和商户管理等接口进一步检查业务级越权。
286	Normal	点餐功能测试从商户切换开始，依次检查分类展示、菜品浏览、购物车增减、地址选择和订单提交。预期结果是：当前商户变化后，页面展示和购物车数据同步变化；同一用户在不同商户下的购物车互不干扰；提交订单后只清理当前商户购物车；订单详情能够显示商品明细和商户信息。该流程覆盖了学生端最核心的业务闭环。
287	Normal	管理端功能测试包括商户管理、菜品管理、分类管理、套餐管理、订单处理和报表查看。商户管理接口应支持新增、修改、分页、详情、状态和营业状态调整；订单接口应支持查询、确认、拒单、派送、完成和取消；报表接口应返回营业额、订单数量和销量排行等数据。测试重点是接口返回结果与当前账号权限范围是否一致。
288	Normal	功能测试结果以“通过、部分通过或受环境限制”记录。对登录、购物车、订单和统计等路径，应同时记录输入数据、操作步骤、实际结果和证据来源。
289	Heading 2	6.3 关键流程验证
290	Normal	订单提交流程验证关注数据写入顺序和异常处理。测试时先构造含有商品的购物车，再提交订单。若地址不存在，应返回业务异常；若购物车为空，应拒绝下单；若商户处于不可下单状态，应阻止订单生成；若订单生成成功，应同时存在订单主表和订单明细，并清理当前商户购物车。该流程验证体现了事务性业务的基本要求。
291	Normal	多商户隔离验证关注同一用户在不同商户下的购物车和订单是否互不影响。测试时分别在商户 A 和商户 B 加入商品，查询购物车时应只返回当前商户数据；提交商户 A 订单后，商户 B 购物车不应被清理。管理端查询订单时，商户员工应只能看到所属商户订单，平台管理员可根据权限查看更大范围数据。
292	Normal	订单通知验证关注 WebSocket 消息是否在关键状态变化时触发。支付成功后，系统应发送新订单消息；用户催单后，系统应发送催单消息。若当前环境不具备稳定 WebSocket 连接，测试应记录为通知链路受环境影响，而不能把未收到消息简单归因于业务逻辑错误。该处理方式能更准确地区分功能缺陷和测试环境限制。
293	Normal	关键流程验证比单接口测试更关注前后状态是否一致。订单提交、购物车清理和通知推送都需要跨多个模块观察结果，因此更能发现数据边界问题。
294	Heading 2	6.4 质量分析
295	Normal	从功能适合性看，系统覆盖了校园外卖点餐的主要业务流程，包括用户浏览、购物车、订单、支付状态、商户处理和运营统计。与需求模型相比，系统已经具备多角色、多端协作和多商户作用域控制的基础能力。从页面结构和流程设计看，小程序端操作路径较清晰，用户可以围绕商户、商品、购物车和订单完成连续操作；这一判断来自页面与流程检查，不等同于正式用户可用性实验。管理端接口封装集中，便于页面模块复用。
296	Normal	从可靠性看，系统在订单提交、购物车清理和订单状态更新中需要保持事务一致性。现有服务层已经将订单与购物车等数据组织在同一业务流程中，但 WebSocket 通知的可靠性仍受连接状态影响。若后续需要更高可靠性，可引入消息队列、消息确认和失败重试机制。质量模型中的可靠性和可维护性要求提示，系统不应只关注功能完成，还应关注异常路径和后续演进[20]。
297	Normal	从安全性看，JWT、账号类型和商户作用域字段为访问控制提供了基础。个人地址和订单信息属于敏感业务数据，应避免跨用户和跨商户访问。系统还需持续检查各 Mapper 查询是否都带有必要的商户过滤条件，以及管理端接口是否区分平台管理员和商户管理员权限。安全性分析不等同于正式渗透测试，但可以为后续测试提供重点。
298	Normal	从页面结构和流程设计看，小程序端操作路径较清晰；这一判断来自页面结构检查，而不是正式用户可用性实验。后续若开展用户测试，可进一步量化学习成本和满意度。
299	Normal	质量分析还需要区分已经验证的质量和推测的质量。功能适合性、商户边界、购物车隔离和订单流转可以通过流程检查获得证据；并发性能、长期稳定性和用户满意度则需要专门测试或调研才能下结论。将两类质量分开表述，可以减少论文中常见的空泛结论和过度承诺。
