## 3. 论文升级范围

### 3.1 摘要与绪论

摘要、关键词和绪论部分需要修正过度或不准确的技术表述。重点包括：

- 将系统架构表述为 Spring Boot 分层单体后端、uni-app 微信小程序、Nginx 静态 Web 管理端以及 MySQL/Redis/WebSocket 等支撑组件。
- 避免把系统写成“微服务架构”，除非明确说明只是具备模块化或微服务启发式的设计思想。
- 避免把多商户能力表述为“schema 隔离”，应改为基于 `merchant_id` 的共享表行级商户作用域控制。
- 对真实支付、正式压测、用户调研和生产级通知可靠性继续保持边界说明。

### 3.2 第二章：相关研究与技术基础

第二章重点补强技术选型和适配性分析，避免只罗列技术名词。需要补充：

- Spring Boot + MyBatis 适宜当前毕业设计规模和 SQL 可控性。
- uni-app 适宜微信小程序场景，并且和现有代码结构相契合。
- JWT 适合无状态接口认证，但当前论文只说明基础 token 鉴权和过期边界，不扩展为完整身份治理体系。
- WebSocket 适合订单提醒类实时通知，同时说明其连接管理、消息确认和离线补偿仍是后续改进方向。
- Redis 的作用要写得稳妥，说明其在缓存或状态辅助中的位置，不把它写成已经具备完整生产级缓存治理。
- 对微服务、分库分表、独立 schema、多租户数据库等方案作对比，说明当前系统选择共享表行级隔离的主要缘由是校园商户规模可控、部署成本低、实现复杂度适宜。

### 3.3 第四章：架构与数据模型

第四章是重点升级章节，需要新增或补强以下内容：

- 系统总体架构图，展示小程序端、Web 管理端、Nginx、Spring Boot 服务、MySQL、Redis、WebSocket 的关系。
- 部署结构图，说明静态资源、后端服务、数据库和缓存组件的部署关系。
- 核心 ER 图，展示 user、merchant、employee、category、dish、shopping_cart、orders、order_detail 等实体关系。
- 角色权限矩阵，区分学生用户、商户员工、商户管理员、平台管理员的操作范围。
- 将多商户隔离表述为“共享表中的行级作用域控制”，说明 `merchant_id`、`campus_id`、`account_type` 和服务层校验共同构成访问边界。
- 补充兼容模式说明：若部署环境缺少多商户迁移字段，系统存在单店兼容回落，论文结论不得泛化到所有数据库状态。

### 3.4 第五章：关键业务流程实现

第五章保留原有 5.1 至 5.9 的标题结构，不进行大规模章节合并。升级方向是减少重复表达，并增强每个流程与代码证据的对应关系：

- 登录认证：说明用户端 `authentication` 和管理端 `token` 的不同请求头，以及 JWT 中身份、账号类型和商户上下文的作用。
- 商户切换：说明 `merchantId` 和 `shopId` 的兼容处理，以及前后端如何维持当前商户上下文。
- 菜品浏览：说明分类、菜品、套餐数据如何按照当前商户范围展示。
- 购物车隔离：重点说明 `user_id + merchant_id` 的联合边界，以及提交订单后只清理当前商户购物车。
- 订单提交：说明订单主表、订单明细、地址、购物车清理和事务一致性的业务关系。
- 支付边界：继续区分模拟支付状态流转和生产级微信支付上线能力。
- 订单通知：说明 WebSocket 新订单和催单消息的触发点，但不夸大为具备完整 ACK/重试/离线补偿能力。
- 历史订单与再来一单：说明订单详情回显和复购路径。
- 运营统计：说明统计口径和商户权限范围，避免把平台口径与商户口径混写。
