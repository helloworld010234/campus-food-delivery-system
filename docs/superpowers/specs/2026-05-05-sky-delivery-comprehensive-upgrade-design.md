# Sky Delivery 项目综合升级工程设计

日期：2026-05-05

## 1. 背景

本设计以 `D:\sky-delivery` 为升级对象，参考 `D:\feynman-test-2026-05-05` 中的测试产物，尤其是三份核心报告：

- `reports/final-test-harness-report.md`
- `reports/matrix.csv`
- `reports/regression-vs-2026-05-04.md`

报告显示，Feynman v0.2.40 对当前项目完成了全表面测试：CLI 基础能力稳定，技能类输出大多能产生可用产物，但存在 REPL-only slash 命令无法通过非交互方式触发、部分技能 TIMEOUT/PARTIAL、以及与 2026-05-04 基线相比若干技能回归。对于 `D:\sky-delivery`，这些报告不直接等价于业务功能失败清单，而是为项目升级提供了证据型输入：哪些工程能力已经可复用，哪些验证方式需要补强，哪些项目说明、部署流程和质量门禁需要系统化。

当前仓库同时存在一个进行中的多商户隔离 Ralph 循环。该专项不再作为唯一主线，而应并入新的综合升级工程，成为后端安全与权限边界升级的一部分。

## 2. 主任务

主任务命名为：**Sky Delivery 项目综合升级工程**。

主任务只升级 `D:\sky-delivery` 项目本身。论文、Feynman 工具测试和研究输出只作为输入材料，不作为本轮项目交付主体。

主任务目标是将校园外卖项目从“功能可运行”推进到“边界清晰、链路可靠、环境可复现、验证可追踪”的状态。升级方向包括后端安全、订单链路、运营后台、小程序体验、部署工程化和验证体系。

## 3. 非目标

本轮综合升级不做以下事项：

- 不把论文 DOCX 升级作为主交付。
- 不修复 Feynman CLI 或 Feynman 技能本体。
- 不把项目重写成微服务。
- 不引入 schema-per-tenant 或 database-per-tenant 的多租户重构。
- 不替换现有 Spring Boot、MyBatis、uni-app、Nginx 技术栈。
- 不要求一次性完成生产级微信支付、骑手调度、消息队列和大规模压测。
- 设计阶段不修改 Java、JavaScript、小程序、脚本或配置文件。
- 设计阶段不启动 6 个 agent，不重建 `scripts/ralph/prd.json`，不运行测试。

## 4. 升级原则

### 4.1 证据驱动

升级应使用 Feynman 报告、源码事实、已有测试和项目文档作为输入。每一条升级线都要记录证据来源、验证命令和残留风险。

### 4.2 保持现有架构

项目继续保持 Spring Boot 分层单体后端、uni-app 小程序、Nginx 静态后台、MySQL、Redis、WebSocket 的总体架构。升级应优先补强边界和验证，不做大范围重写。

### 4.3 多 agent 并发友好

文档需要为 6 个 agent 并发执行做好准备。每个 agent 应有清晰的目标、文件所有权、禁止触碰范围、输入材料、测试要求和交付摘要格式。

### 4.4 共享文件由协调者统一维护

`task_plan.md`、`findings.md`、`progress.md`、`scripts/ralph/progress.txt`、`scripts/ralph/required-gates.md` 等共享文件由主协调者统一更新。单个 agent 不应直接争抢这些文件。

### 4.5 不污染 C 盘

临时文件、生成产物、转换输出、缓存、辅助脚本和中间数据优先放在 `D:\sky-delivery` 或用户指定的非 C 盘路径。只有读取已安装工具、用户源文件或不可移动运行时路径时才使用 C 盘。

## 5. 6-agent 工作面

### Agent 1：后端安全与权限边界

负责 JWT、登出黑名单、平台/商户账号权限、多商户隔离、越权矩阵、服务层 guard、跨商户访问拒绝和公开浏览边界保护。现有多商户隔离 Ralph 工作并入该线。

### Agent 2：订单与交易链路

负责购物车、下单、支付模拟边界、订单状态流转、复购、催单、订单明细一致性、事务边界、跨商户购物车与订单污染防护。

### Agent 3：运营后台与报表工作台

负责后台管理接口、员工/商户/菜品/套餐/分类管理、报表统计口径、工作台指标、平台视角与商户视角隔离、导出能力。

### Agent 4：小程序与用户体验

负责 uni-app 用户端商户选择、菜单浏览、购物车体验、地址管理、历史订单、接口参数一致性、登录态处理、用户侧通知边界。

### Agent 5：部署、环境与工程化

负责启动脚本、环境变量、数据库迁移、Docker/Compose、健康检查、Redis/MySQL 配置、README/SETUP 可复现性、C 盘写入规避。

### Agent 6：验证体系与质量证据

负责 Maven 测试、Playwright E2E、接口回归矩阵、Feynman 报告吸收、TIMEOUT/PARTIAL/SKIP 解释、质量门禁、证据归档和最终升级报告。

## 6. 总 Ralph 故事结构

后续执行时，新的 Ralph 循环建议包含 8 个故事：

| Story | 名称 | 执行主体 |
|---|---|---|
| US-001 | 综合升级控制面与旧 Ralph 迁移 | 主协调者 |
| US-002 | 后端安全与权限边界升级 | Agent 1 |
| US-003 | 订单与交易链路升级 | Agent 2 |
| US-004 | 运营后台与报表工作台升级 | Agent 3 |
| US-005 | 小程序与用户体验升级 | Agent 4 |
| US-006 | 部署、环境与工程化升级 | Agent 5 |
| US-007 | 验证体系与质量证据升级 | Agent 6 |
| US-008 | 六线集成、回归验证与最终项目升级报告 | 主协调者 |

## 7. 并发协作规则

### 7.1 文件所有权

每个 agent 有主写范围。其他 agent 如需修改该范围内文件，应在交付摘要中提出协调请求，由主协调者合并。

### 7.2 接口契约优先

跨 agent 依赖通过接口契约沟通。例如 Agent 2 可以依赖 Agent 1 定义的商户上下文解析语义，但不应私自重写安全 guard。Agent 4 可以依赖后端请求参数契约，但不应绕过后端权限规则。

### 7.3 并行与集成分离

6 个 agent 可并行进行探索、测试设计、局部实现方案和文档准备。最终合并时按下面顺序集成：

1. 后端安全与权限边界
2. 订单与交易链路
3. 运营后台与报表工作台
4. 小程序与用户体验
5. 部署、环境与工程化
6. 验证体系与质量证据

### 7.4 交付摘要格式

每个 agent 完成后必须输出：

- 处理范围
- 修改或建议修改的文件
- 测试命令与结果
- 已通过的验收标准
- 剩余风险
- 需要其他 agent 或协调者处理的事项

## 8. Feynman 报告吸收策略

### 8.1 PASS 与 improved

作为已有能力和可复用测试资产。例如 CLI 基础能力、若干技能输出、deep-research、watch、replication 的改善可作为质量体系输入。

### 8.2 PARTIAL 与 TIMEOUT

不直接视为 Sky Delivery 功能失败，而转化为验证体系改进：任务拆分、超时设置、增量产物保存、可恢复执行和证据分类。

### 8.3 SKIP

作为环境或交互约束记录。比如 REPL-only slash 命令无法通过 `--prompt` 或 `chat <text>` 触发，应记录为测试方法限制，而不是项目业务缺陷。

## 9. 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 6 个 agent 修改同一文件 | 合并冲突、覆盖已有工作 | 通过 brief 明确文件所有权，共享文件由协调者维护 |
| 当前多商户隔离工作被覆盖 | 丢失 US-001/US-002 证据 | 先归档旧 Ralph，并在 Agent 1 中吸收已有进度 |
| 升级范围过大 | 无法一次完成 | 使用 8 个 Ralph stories 与 6 条并行线分批落地 |
| Feynman 报告被误读 | 把工具限制误判为项目缺陷 | 使用报告吸收矩阵区分 PASS/PARTIAL/TIMEOUT/SKIP |
| 部署材料引入敏感信息 | 泄露密钥或环境变量 | 只写模板与变量名，不提交真实 secret |
| 业务升级破坏公开浏览 | 用户端跨商户菜单不可用 | Agent 1、Agent 2、Agent 4、Agent 6 都设置公开浏览回归要求 |

## 10. 验收标准

设计阶段完成后，应满足：

- 总设计文档存在并覆盖主任务、6-agent 模型、Ralph 重建蓝图和门禁。
- 总 PRD 以 Markdown 形式存在，能作为后续 `ralph-convert` 输入。
- 6 份 agent brief 均存在，且每份都包含目标、输入、文件范围、禁止触碰范围、执行内容、执行阶段注意事项、测试和验收标准。
- 并行执行计划、Ralph 蓝图、门禁设计、Feynman 报告吸收文档和最终报告模板均为 Markdown。
- 未修改业务代码，未重建 `scripts/ralph/prd.json`，未启动 agent。

后续执行阶段完成后，应满足：

- 旧 Ralph 已归档，新总 Ralph 已建立。
- 6 条 agent 工作线均有测试证据和交付摘要。
- 后端 Maven 测试、关键 E2E/API 测试和质量矩阵有明确结果。
- 最终项目升级报告完成，列明改动、证据和残留风险。
