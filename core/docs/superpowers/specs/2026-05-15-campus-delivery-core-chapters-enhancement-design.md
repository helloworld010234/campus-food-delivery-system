# 校园外卖论文核心章正文增强设计

日期：2026-05-15

## 1. 目标

本次正文升级聚焦《校园外卖设计与实现》论文第 2 章至第 6 章，目标是把当前偏空泛的正文改成有业务场景、有设计理由、有源码依据、又能让非专业答辩老师理解的毕业论文表达。摘要、绪论和总结只做轻微同步，不做大范围重写。

正文不能直接套入源码内容，也不把源码方法、类名和接口路径机械堆进正文。源码在本次工作中只作为事实证据，用来判断系统真实做了什么；最终行文要经过概括、转译和组织，形成“问题是什么、为什么这样设计、系统怎样解决、运行效果如何验证”的文章思路。

## 2. 范围

重点增强范围：

- 第二章：校园点餐业务场景分析
- 第三章：系统需求分析
- 第四章：系统总体设计
- 第五章：系统详细设计
- 第六章：系统实现与运行效果

轻微同步范围：

- 摘要：根据核心章增强后的主线补充“多商户、订单闭环、角色协同”等表达
- 第一章：保持绪论框架，只修补研究内容与论文组织中和正文不一致的地方
- 第七章：同步总结语气，强调系统完成的业务闭环和后续可扩展点

不处理范围：

- 不新增未经验证的功能
- 不直接粘贴源码
- 不把正文改成接口清单或代码说明书
- 不改动图片风格和截图，除非后续实施中发现正文引用明显不一致

## 3. 证据来源

正文增强将以项目源码和已有论文素材为证据基础，主要参考：

- 后端 Spring Boot 工程：`backend/sky-server`、`backend/sky-common`、`backend/sky-pojo`
- 数据库脚本：`database/init.sql`、`backend/scripts/phase1_multi_merchant_schema.sql`
- 小程序端：`miniapp/pages`、`miniapp/pages/api/api.js`
- Web 管理端：`nginx/html/sky/merchant-admin/assets/js`
- 已处理的 DOCX：上一轮生成的“干净截图版”论文文档

源码证据的使用方式是“提炼设计含义”，例如：

- 购物车服务体现的是“同一学生在不同商户下购物车不能混用”
- 订单服务体现的是“订单提交、支付、接单、配送、完成必须按状态推进”
- 商户权限守卫体现的是“平台账号和商家账号的管理范围不同”
- 多商户迁移脚本体现的是“从单店模式扩展到校园多商户模式”
- 小程序页面和接口调用体现的是“学生端完成选商户、选商品、结算、查订单的连续路径”

这些内容进入正文时，应转写为业务设计和系统设计语言，而不是源码讲解。

## 4. 叙事主线

核心章采用“校园场景驱动的多角色点餐系统”作为主线。论文不只说明做了一个点餐系统，而是说明系统如何解决校园场景中的三个实际问题：

1. 学生侧：线下排队、信息不透明、不同商户之间切换不方便
2. 商家侧：订单需要按状态处理，不能只靠人工记忆和口头确认
3. 平台侧：校园和商户需要统一维护，多个商户之间的数据不能混在一起

围绕这条主线，正文应强调三个设计关键词：

- 角色分工：学生、商家、平台三类角色各自负责不同事情
- 数据归属：购物车、商品、订单、统计都需要明确属于哪个用户和哪个商户
- 状态流转：订单从提交到支付、接单、配送、完成，需要一步一步推进

## 5. 分章设计

### 5.1 第二章：校园点餐业务场景分析

第二章要从真实校园场景出发，避免只写“互联网外卖发展迅速”这类泛泛背景。建议把内容组织为：

- 校园用餐集中在课间、午饭和晚饭时段，短时间内订单集中，线下窗口容易排队
- 学生更关心“现在有哪些商户、能不能下单、多久送到、订单到哪一步”
- 商家更关心“哪些订单需要处理、是否已经支付、是否能接单、什么时候完成配送”
- 平台更关心“哪些商户在营业、哪些商户可被学生看到、基础数据是否统一”

写法上要通俗。例如解释多商户时，可以写成“同一校园中可能有多个餐饮商户，学生在一家店选择的商品，不能在另一家店结算；商家也不能看到其他商户的订单”。这比直接说“实现 merchant_id 数据隔离”更适合答辩阅读。

### 5.2 第三章：系统需求分析

第三章从功能列表升级为“角色行为需求”。每一类需求都按“角色目标 + 系统响应 + 约束条件”来写。

学生端需求重点：

- 登录后进入点餐路径
- 选择商户并浏览商品和分类
- 添加购物车、调整数量、清空购物车
- 选择地址并提交订单
- 支付后查看订单状态，必要时催单或再来一单

商家端需求重点：

- 登录后查看本商户范围内数据
- 维护商品、分类、套餐等基础信息
- 处理待接单订单，完成接单、拒单、配送、完成等操作
- 查看订单统计和营业数据

平台端需求重点：

- 维护商户基础资料
- 管理商户启用状态和营业状态
- 查看平台范围内的商户概览和经营统计

非功能需求要避免空话，应结合源码主线写成可理解的设计约束：

- 数据隔离：商家只能处理自己范围内的数据
- 操作可靠：订单状态不能随意跳转
- 信息及时：支付成功和催单后需要通知相关管理端
- 兼容演示：支付可使用模拟支付分支，便于本地运行和答辩展示

### 5.3 第四章：系统总体设计

第四章要解释系统为什么拆成小程序端、Web 管理端、后端服务和数据库。

建议表达：

- 小程序端负责学生的轻量操作，适合手机上完成选购和查看订单
- Web 管理端负责商家和平台人员的集中管理，适合桌面端展示表格、统计和批量操作
- 后端服务负责判断业务规则，例如账号是否有效、商户是否可用、订单状态是否允许变更
- 数据库负责保存长期数据，例如用户、地址、商户、商品、购物车、订单和订单明细

架构表达应少用晦涩术语，多用“谁负责什么”。例如可以解释“前端不能直接改数据库，因为订单状态和商户权限需要由后端统一判断，否则容易出现越权处理订单或订单状态混乱”。

### 5.4 第五章：系统详细设计

第五章是本次增强的重点，应把数据库和流程设计讲成业务逻辑。

数据库设计要围绕关系讲：

- 用户与地址：一个学生可以保存多个收货地址
- 商户与商品/分类：商品必须归属于具体商户和分类
- 购物车：购物车不仅属于用户，也要对应商户，避免跨商户混单
- 订单与订单明细：订单保存整体信息，订单明细保存具体商品
- 校园与商户：平台可以维护校园下的多个商户

订单状态设计要讲清楚“状态是系统的秩序”。例如：

- 未支付订单不能直接进入配送
- 已支付订单进入待接单
- 商家接单后才能进入后续配送和完成
- 拒单或取消需要记录原因，并根据支付状态处理退款或状态调整

支付回调设计要讲成“外部支付结果如何进入系统”。不需要展开支付平台细节，重点说明系统接到支付成功结果后，会把订单从待支付转为待接单，并通知商家端有新订单。

安全与异常处理要围绕“防止错误操作”讲：

- 登录令牌用于识别当前用户或管理员
- 商户范围校验用于防止商家访问其他商户数据
- 订单重复提交锁用于降低重复下单风险
- 全局异常处理用于统一返回错误信息

### 5.5 第六章：系统实现与运行效果

第六章需要结合源码总结实现逻辑，但正文写作仍以流程说明为主。

建议每个小节采用三段式：

1. 先说明该模块解决什么业务问题
2. 再说明系统处理流程
3. 最后结合截图或流程图说明运行效果

重点小节写法：

- 用户登录：说明学生端和管理端都需要身份识别，但身份来源和角色范围不同
- 商户选择与商品浏览：说明先选商户，再读该商户的分类和商品，保证页面数据来源明确
- 添加购物车：说明系统会先判断商品是否属于当前商户，再决定新增记录或累加数量
- 订单提交与支付：说明订单不是直接保存一个总价，而是要校验地址、商户状态、购物车、金额，再生成订单主表和明细
- 支付回调：说明支付结果会改变订单状态，并触发新订单通知
- 商家订单处理：说明商家操作受订单状态限制，不能随意跳过步骤
- 平台管理：说明平台端负责商户基础数据和运营状态，而不是处理单个订单

第六章要避免“代码级口吻”，可以使用“系统会”“服务端会”“管理端页面展示”等表达。

## 6. 文风要求

正文要兼顾学术性和可理解性：

- 句子不要过长，尽量一段讲一个观点
- 少用无法解释的技术术语，必须使用时要顺手解释
- 避免“本系统实现了丰富功能、具有良好扩展性”等空泛句
- 多使用校园点餐中的具体场景来承接技术设计
- 不写“源码中某某方法”这种生硬表达
- 不出现旧项目名、课程模板名或其它与当前论文不一致的系统称呼

推荐表达方式：

- “为了避免学生在不同商户之间混合结算，系统在购物车和订单数据中保留商户归属。”
- “订单状态相当于系统中的业务顺序表，只有满足当前状态要求，下一步操作才会被允许。”
- “平台端主要维护商户是否可见、是否营业等基础状态，这些状态会影响学生端能否下单。”

不推荐表达方式：

- 直接写某个服务类调用某个数据访问方法。
- 直接写某个方法内部先执行了某个技术步骤。
- 直接把权限判断类、接口路径或数据库字段当作正文主体。

## 7. 实施策略

实施时不直接在 DOCX 中盲改，应先从现有 DOCX 提取正文结构，形成章节级草稿，再回填到 Word 文档。每章修改时保持原有标题和图表位置，尽量不改变图片版面。

建议实施顺序：

1. 提取第 2-6 章原文，建立段落替换清单
2. 按源码证据整理每章事实点
3. 生成第 2-6 章增强稿
4. 轻微同步摘要、绪论和总结
5. 回写 DOCX
6. 渲染 DOCX 页面 PNG 检查版面，不导出 PDF
7. 检查禁用词、图题连续性、章节标题和图片是否保留

## 8. 成功标准

本次正文升级完成后，应满足：

- 第 2-6 章明显不再空泛，每章都有清楚的设计逻辑
- 正文能让非专业老师理解系统为什么这样设计
- 关键设计均能追溯到真实源码或数据库结构
- 没有直接粘贴源码、方法名堆叠或接口清单式写法
- 不改变论文原有格式框架
- 不破坏已有图表和干净截图
- 不生成 PDF

## 9. 风险与控制

风险一：内容过度技术化。控制方式是以“业务问题”开头，以“设计作用”收束，源码只在内部作为证据。

风险二：扩写导致版面过长。控制方式是控制每节新增内容密度，第 2-6 章重点增强，摘要、绪论和总结只轻微同步。

风险三：引入源码不存在的功能。控制方式是每个关键说法都先在源码、数据库脚本、小程序页面或管理端页面中找到依据。

风险四：图文不一致。控制方式是改写时保留现有图题和图片位置，并让正文围绕已存在的流程图、ER 图和截图解释。

- DOCX 回写与页面 PNG 渲染检查
