# 校园外卖论文图表与截图完整增强设计规格

日期：2026-05-14  
状态：待用户审阅  
适用目标：对《校园外卖设计与实现》毕业论文进行图片体系升级，重点提升图表完整性、截图真实性、后端设计表达和论文版面观感。

## 1. 背景与目标

上一轮论文图片存在表达过简、流程分支不足、截图来源混用、后端与数据库图表支撑不充分等问题。本轮采用“完整增强版”路线，围绕源码、数据库、业务流程和真实界面四类证据重新组织图片体系。

本次升级的核心目标如下：

- 主文和附录都明显增强图片证据，优先保证完整性和论文说服力。
- 小程序端截图必须从用户提供的原论文 DOCX 中提取并复用，不使用项目目录中的其他设计稿替代。
- Web 管理端截图采用“两者结合”：先复用已有真实截图，不足页面再启动系统补拍。
- 后端流程图、数据库 ER 图、架构图和模块图全部重绘，达到可直接插入 Word 的学术论文图表质量。
- 流程图保持原子化，但每张图必须完整描述一个闭环逻辑，包含正常路径、失败路径、关键判断和结束节点。
- 不改变学校论文模板、标题层级、页眉页脚、目录样式、正文格式等既有格式。

## 2. 成功标准

本轮完成后应满足以下标准：

- 论文主文形成“架构设计 -> 数据库设计 -> 核心流程实现 -> 界面效果验证”的连续证据链。
- 每张流程图都使用垂直布局，普通步骤为矩形，判断节点为菱形，并清楚标注“是/否”“成功/失败”等分支。
- 每张流程图只聚焦一个子流程，但不省略关键异常逻辑；例如支付回调必须包含签名校验、订单查询、重复回调、金额校验、状态更新和失败返回。
- ER 图标注主键、外键、主要属性和实体关系基数，包括 1:1、1:N、N:M。
- 图片风格统一为中文标签、黑白灰主色、深蓝强调色，避免艳丽配色和花哨装饰。
- 主文图片优先控制在半页以内，复杂图拆分为“主文简图 + 附录详图”。
- 主文避免一页只有一张图片；避免连续堆放两张图片，中间需有正文解释或段落过渡。
- 所有图片输出 SVG 或 300DPI 以上高清 PNG，并保留原始素材与处理后图片。
- 论文图片、标题、正文引用中不得出现项目历史名称、演示库名或与学校论文不相符的培训项目字样。

## 3. 素材来源策略

### 3.1 小程序端截图

小程序端截图只从用户提供的原论文 DOCX 中提取。实施时从 DOCX 的 `word/media` 中提取图片，并结合图片比例、论文位置和截图内容识别学生端页面。

处理规则：

- 保留原截图的真实界面内容，不改动业务文本和页面元素。
- 只进行裁剪、边距统一、白底排版、编号标注、细边框和轻微清晰化处理。
- 优先使用两图并排或三图并排组合方式，例如“首页 / 商品浏览 / 下单确认”。
- 对于竖向手机截图，正文中不单张铺满一页；附录可保留更完整截图组。
- 每组图下方标题使用“图 6-x 小程序端页面效果图”这类规范表达。

### 3.2 Web 管理端截图

Web 管理端采用“复用已有截图 + 缺口补拍”的组合策略。

优先复用已存在的真实截图：

- 首页看板页面。
- 商品管理页面。
- 订单管理页面。
- 统计分析页面。
- 已处理的管理端组合截图。

若缺少关键页面或截图质量不足，再使用桌面启动脚本启动系统补拍。补拍时使用真实浏览器页面，登录管理员账号后截取应用主体区域，避免浏览器边框、地址栏和桌面背景进入论文图片。

处理规则：

- 截图宽度以 1440px 左右为基准，适合高清裁剪。
- 长页面优先截核心区域，不把超长整页强行压缩到正文。
- 采用统一细边框、浅灰底、深蓝编号和箭头标注。
- 主文优先展示功能代表性最强的页面，附录展示更完整截图组。

### 3.3 后端与数据库图表

后端流程、数据库、系统架构和模块图全部重绘。绘制依据来自当前源码、数据库表结构、接口模块和论文已有描述。

处理规则：

- 系统架构、模块结构、数据库关系等复杂图优先使用 SVG 重绘，以获得更好的版面控制。
- 原子流程图可使用 Mermaid 或 Graphviz 生成 SVG，再统一转为高清 PNG。
- 每张图保留可编辑源文件，便于修改节点或标题。
- 所有节点使用中文标签；技术组件可保留必要英文技术名，例如 Spring Boot、MySQL、Redis、JWT、WebSocket，但必须服务于论文表达。

## 6. 流程图完整性规范

所有流程图必须遵守以下结构：

1. 开始节点：说明触发入口，例如“用户提交登录请求”。
2. 输入与前置条件：体现关键参数、登录态、店铺状态、购物车状态等。
3. 校验节点：使用菱形表示判断，分支必须标注“是/否”。
4. 正常路径：描述从请求到数据处理再到响应的完整链路。
5. 异常路径：至少覆盖参数错误、状态不合法、数据不存在、重复请求、外部服务失败或数据库更新失败中的相关情况。
6. 结果节点：正常结束和异常结束都必须收束。

各核心流程的最低分支要求如下：

| 流程图 | 必含分支 |
| --- | --- |
| 用户登录认证流程图 | 登录凭证有效/无效，用户存在/不存在，令牌生成成功/失败 |
| 商品浏览与规格选择流程图 | 店铺营业/休息，商品上架/下架，规格已选/未选，商品可售/不可售 |
| 添加购物车流程图 | 购物车已有该商品/没有该商品，商品有效/无效，更新成功/失败 |
| 订单提交流程图 | 地址有效/无效，购物车为空/非空，金额校验通过/失败，订单创建成功/失败 |
| 支付与回调处理流程图 | 签名有效/无效，订单存在/不存在，金额一致/不一致，重复回调/首次回调，状态更新成功/失败 |
| 商家接单与出餐流程图 | 接单/拒单，订单状态可处理/不可处理，通知成功/失败，超时或取消分支 |
| 管理端商品维护流程图 | 新增/修改/上下架，分类有效/无效，图片上传成功/失败，保存成功/失败 |

流程图节点数量以 12 至 18 个为目标。若一个流程超过该范围但逻辑不可再压缩，允许扩展到 20 个以内；超过后应拆分为两个图，例如“订单提交前校验”和“订单创建事务处理”。

## 7. 图表视觉规范

统一视觉标准如下：

- 画布背景：白色。
- 主色：黑、白、灰。
- 强调色：深蓝色，用于关键模块边框、状态节点或编号。
- 字体：中文优先使用宋体、黑体或兼容字体；导出时确保字体嵌入或转曲。
- 线条：1.2pt 至 1.8pt，箭头清晰但不过粗。
- 节点：矩形、圆角矩形、菱形、数据库圆柱等均按学术图表标准使用。
- 边框：浅灰或深灰，重点节点使用深蓝边框。
- 禁止：艳丽渐变、装饰性图标堆叠、复杂阴影、大面积彩色块、与论文主题无关的视觉素材。

截图排版规范如下：

- Web 截图使用统一外框和标题编号。
- 手机截图组合图使用等高排列，子图底部或顶部标注“a / b / c”。
- 箭头与编号只标注关键功能点，避免遮挡界面内容。
- 图片压缩后仍需能读清页面主标题、按钮和表格列名。

## 8. 论文引用与版面策略

主文中每张图都应有明确引导语，避免图片孤立出现。推荐正文引用方式：

- “系统采用前后端分离架构，整体部署与调用关系如图 3-1 所示。”
- “订单相关实体之间的联系较为紧密，其领域关系如图 4-2 所示。”
- “为保证支付结果可靠入库，系统在回调阶段增加签名校验、金额校验和幂等处理，流程如图 5-5 所示。”
- “管理端订单处理页面如图 5-9 所示，管理员可按订单状态进行筛选和处理。”
- “小程序端页面效果如图 6-1 和图 6-2 所示，截图来源于原论文中的实际运行效果图。”

版面控制策略：

- 主文单张流程图宽度控制在页面正文宽度的 80% 至 95%。
- 组合截图优先使用横向拼接，减少竖向占页。
- ER 总图如无法在主文保持清晰，则主文使用简化图，完整图放附录。
- 同一页出现两张图片时，需保证每张图上下有简短说明；避免连续图片无文字过渡。
- 若图片导致分页不美观，优先调整图宽、组合方式或迁移到附录，不破坏学校模板。

## 9. 产物目录设计

实施阶段的建议产物目录如下：

```text
paper-output/figure-upgrade/
  figures/
    sources/       # Mermaid、Graphviz、SVG 源文件
    svg/           # 可编辑矢量图
    png/           # 论文插入版高清图片
  screenshots/
    extracted/     # 从原论文 DOCX 提取的原始图片
    web-original/  # 复用或补拍的 Web 原始截图
    processed/     # 已裁剪、标注、排版后的截图
  reports/
    figure-registry.json
    image-source-map.json
    static-qa.json
```

`figure-registry.json` 记录每张图的标题、来源、章节、插入位置、尺寸、文件路径和是否进入附录。`image-source-map.json` 记录小程序截图来自原论文 DOCX 的哪一个嵌入图片，Web 截图来自已有截图还是补拍。

## 10. 验证与质量检查

实施完成前必须执行以下检查：

- 图片来源检查：小程序截图均来自用户提供的原论文 DOCX。
- 禁用词检查：论文正文、图题、图中文字和导出文件名不得包含项目历史名称、演示库名或培训项目名称。
- 图表完整性检查：每张流程图有开始、判断、正常路径、失败路径和结束节点。
- ER 图检查：主要实体有 PK/FK 标识，关系基数清晰。
- 分辨率检查：PNG 图片满足 300DPI 以上插入需求；SVG 可正常打开。
- 版面检查：主文不出现一页一图或连续堆图的观感；附录图片标题连续、尺寸统一。
- DOCX 渲染检查：渲染后抽查页面，确认图片不变形、不遮挡、不跨页断裂。
- 源码一致性检查：后端流程和数据库图表与当前源码、数据库结构、接口模块相符。

## 11. 非目标

本轮不做以下事项：

- 不重新设计小程序界面。
- 不把项目目录中的小程序设计稿冒充为论文小程序截图。
- 不改变学校模板格式、页眉页脚、目录样式和已有论文格式规范。
- 不修改业务源码逻辑，除非后续用户明确要求修复系统问题。
- 不把所有截图强行塞入主文；完整截图组优先放附录。
